# Cursor 复盘：Agent Harness 不是设计完就完了，是要持续运营的

Cursor 最近发了一篇《Continually improving our agent harness》，表面在讲他们怎么改进 Agent，实际在回答一个更根本的问题：**当 Harness 开始承重，团队怎么持续运营它？**

## 核心观点

**发布单元不是模型，是"模型 + Harness"的组合。** 同一个模型放进不同 Harness，体感差距巨大——GPT-5.5 在 Cursor Harness 里比原生表现好，Claude Opus 4.7 从 Claude Code 换到 Cursor Harness 功能性从 87.2% 跳到 91.1%。

**上下文管理从"提前堆砌"转向"现场获取"。** 2024 年末 Cursor 做法是每次编辑后回灌 lint 错误、替模型改写读取请求、会话开始就塞目录结构和代码片段。现在模型能力变强，这些旧护栏反而变成限速带，需要定期清理。

**工具可靠性 = Agent 质量。** 工具错误不只是"这一步失败"，错误信息会留在上下文里，污染后续所有决策。所以 tool reliability 要按模型和工具维度拆开监控。

**评测不能只看 benchmark。** CursorBench 做离线评测，但真实质量还要看线上 A/B、Keep Rate（用户是否保留 Agent 的改动）、用户后续回复的语义分析。

**多模型适配是真实工程负担。** OpenAI、Anthropic 对工具格式、提示词、消息顺序、缓存策略的反应都不一样。用户中途切模型会带来缓存失效和历史分布错位。

## 为什么值得关注

这篇文章的价值不在于提出新概念，而在于把 Agent Harness 的运营拉到了 SRE 级别的工程纪律：失败要留结构化痕迹，退化要能切片定位，模型升级要重检旧规则，每次改动都有假设-实验-指标-观测-回滚的闭环。

作者（架构师公众号）把这条线串得很清楚：从"模型差距在缩小 Harness 差距在放大"→"Agent Harness 综述"→"Harness 正在成为新后端"→ 这篇"Harness 怎么被长期维护"。是一个完整的认知递进。
