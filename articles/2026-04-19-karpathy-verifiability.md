# Karpathy: 可验证性是 AI 应用的分水岭

> 来源：Andrej Karpathy 推文/演讲
> 标签：AI, Develop

## 核心论点

Andrej Karpathy 提出了一个简洁有力的框架来判断 AI 在哪些领域会最先成功：看输出的可验证性。

可验证性高的任务——代码（能编译、能跑测试）、数学（能验算）、棋类（有明确胜负）——AI 进步最快，因为有清晰的反馈信号。

可验证性低的任务——写小说（好坏主观）、做设计（品味因人而异）、制定战略（结果要很久才知道）——AI 进步慢，因为缺乏可靠的训练信号。

## 为什么这个框架有用

它解释了很多现象：

**为什么 coding agent 比 writing agent 好用？** 代码有编译器和测试套件做验证器，写作没有。coding agent 可以写代码→跑测试→看结果→修改，形成闭环。writing agent 写完只能问人"这样行吗？"

**为什么 AlphaGo 比自动驾驶先成功？** 围棋规则完全确定，每步都可验证。驾驶场景无限多样，"安全"的定义模糊。

**为什么 RLHF 有效但有上限？** 人类反馈是一种验证信号，但噪音大、成本高、不一致。比起编译器的确定性反馈，RLHF 的信号质量差很多。

## 对 Agent 工程的启示

设计 agent 系统时，首先要问：这个任务的输出能自动验证吗？

- **能自动验证** → 让 agent 自主循环（写代码→测试→修复），人类只做最终 review
- **部分可验证** → agent 做初稿，人类验证关键决策点
- **不可验证** → agent 辅助（提供选项、搜集信息），人类做决策

Anthropic 的 long-running harness 用 Puppeteer 做浏览器自动化测试，就是在给 coding agent 增加验证能力。agent 不只是写代码，还能打开浏览器看效果、点按钮测交互。验证能力越强，agent 的自主度越高。

## 跟我们的关联

我们四角色 agent 团队里的"测试验收 agent"本质上就是一个验证器。它的存在让执行 agent 的输出变得可验证——不是靠人类逐行 review，而是靠自动化测试 + 截图对照。

board CLI 的 GATE 环节也是验证点。但目前 GATE 是人工的（kenefe 验收），未来可以加入更多自动化验证：lint 通过、测试通过、截图对比无差异、性能基准不退化。

可验证性越高 → agent 自主度越高 → 人类介入越少 → 效率越高。提升 agent 效率的关键不是让模型更聪明，而是让验证更自动化。
