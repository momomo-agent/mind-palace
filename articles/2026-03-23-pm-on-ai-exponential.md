# AI 指数增长下的产品管理：Claude Code 产品负责人的方法论

Cat Wu 是 Anthropic Claude Code 的产品负责人。这篇文章是她对"AI 指数增长如何改变产品管理"的系统思考——不是空泛的趋势预测，而是来自 Claude Code 团队日常实践的第一手经验。

## 指数增长的实感

Cat Wu 从 2024 年开始养成一个习惯：每次新模型发布，都让 Claude Code 给 Excalidraw 加一个表格工具。Sonnet 3.5 做不到，Opus 4 偶尔成功（成功率够做录播 demo），Opus 4.6 稳定到可以现场直播演示。

METR 的数据更直观：Opus 4.6 能完成人类需要 12 小时的软件任务。16 个月前 Sonnet 3.5 只能做 21 分钟的。**16 个月，41 倍跳跃。**

这个速度打破了传统产品管理的根本假设——"项目开始时技术能做什么，结束时大致也能做什么"。当模型能力以指数增长，你为限制做的设计可能在项目中途就变成多余的。你是在不断上升的地面上建东西。

## 三工具工作流

Cat Wu 的日常分工很清晰：

- **Claude.ai** — 思考伙伴。讨论策略、处理棘手情况、快速问答。不需要它动手，只需要它思考。
- **Claude Code** — 写代码。原型、eval 脚本、调 API 的工具。输出是代码的事都在这里。
- **Cowork** — 其他一切。收件箱清零、todo 管理、做 PPT、搜 Slack 历史、订差旅。

这个分工的本质是按"输出类型"划分 AI 工具——思考、代码、行动。

## 四个核心转变

### 1. Side Quests 替代长期路线图

传统做法：调研→锁定路线图→按计划执行。

Claude Code 团队的做法：鼓励所有人（工程师、设计师、PM）花一个下午做 side quest——在正式路线图之外的自发实验。测试一个你以为还做不到的能力，看看推模型更用力会发生什么。

结果：Claude Code Desktop、AskUserQuestion 工具、todo lists——这些最受欢迎的功能全是 side quest 产出的，不是路线图规划的。

### 2. Demo + Eval 替代文档

传统做法：写 PRD→评审→交给工程团队→build。

现在的做法：写完 spec 直接扔给 Claude Code 看能不能 build。粗糙原型比精美文档更能推动决策。

Cat Wu 给了一个具体例子：Noah 写了 plugins 的 spec 交给 Claude Code，回来的原型接近生产可用。这个原型成了最终发布功能的锚点——因为团队能立刻验证 UX 是否成立。

**Pro-tip**：写完任何 spec，先让 Claude Code 试着做。即使粗糙的原型也能改变整个对话。

对于不好原型化的功能（比如 Agent Teams），用 eval 替代。手工构建测试用例来理解功能什么时候好用、什么时候不好用、该修什么。能测量就能改进。

### 3. 每次模型升级重审已有功能

传统产品发布后进入维护模式。但在 AI 产品里，每次新模型发布都是一个隐含的提示：重新审视你已经做的东西。

Claude Code with Chrome 就是这样来的。团队注意到用户在 Claude Code 里写 web app 后手动切到 Chrome 测试，来回复制粘贴。用户自己在 hack 的东西，就是你应该内建的功能。

关键原则：**先优化能力，后优化成本。** 先用比你觉得必要的更多 token。过早砍 token 会让功能大打折扣。等便宜模型追上来再降成本，但首先你得知道这个功能是否可能。

### 4. 做最简单的事

todo lists 刚上线时模型不能可靠地勾完成项。团队加了系统提醒——每隔几条消息提醒 agent 更新 todo list。这是个 hack。下一代模型发布后，行为自然就有了，提醒直接删掉。

这个模式反复出现：system prompt 和 tool descriptions 曾经为了补偿模型限制而精心设计。每次模型升级，都在砍 prompt。**Opus 4.6 的 system prompt 比上一版砍了 20%。**

越简单的实现，越容易吃到模型升级红利。越复杂的 workaround，越可能变成下一代的技术债。

## 角色融合

Claude Code 团队的角色界限在消融：设计师提交代码，工程师做产品决策，PM 做原型和 eval。

这能运转的前提是：**清晰的战略和目标**让每个人都能自主排优先级。PM 的工作是在快速模型进步制造的不确定性中创造清晰度，推动团队往更大的可能性思考，清除发布障碍。

Cat Wu 说自己最难适应的转变是"放手"——作为完美主义者习惯控制全部产品体验。但现在 PM 的角色是识别少数真正的 non-negotiable，然后放手其他一切。

## 对我们的启发

这篇文章验证了我们已经在做的几件事：

1. **kenefe 的四角色 agent 团队**和 Cat Wu 的"角色融合"是同一个趋势的两面——团队边界模糊，每个人都做全栈
2. **side quest 模式**跟我们的"先做再说"风格一致——api2cli instant mode 就是一个 2 小时的 side quest
3. **system prompt 持续瘦身**是我们自己也在经历的——记忆系统的 prompt engineering 随模型升级在减少
4. **先能力后成本**跟 kenefe 反复说的"先让它能跑，再优化"完全一致

但有一个关键差异值得注意：Cat Wu 说"PM's craft has shifted from defining certainty upfront to accelerating discovery"——PM 的核心从"事前定义确定性"变成了"加速发现"。这跟 Datadog PM Kai Xin Tai 说的一样："不是预先定义，而是加速发现。"

**在模型能力指数增长的世界里，发现速度比规划质量更重要。** 因为你规划的约束可能半个月后就不存在了。
