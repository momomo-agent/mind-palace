# Talker-Reasoner: 快思考与慢思考的 Agent 架构

> 来源：Google DeepMind 论文
> 标签：AI, Develop

## Kahneman 的双系统理论用于 Agent

Daniel Kahneman 在《思考，快与慢》里提出人脑有两个系统：System 1 快速直觉，System 2 慢速深思。Google DeepMind 把这个认知科学框架直接映射到了 agent 架构上。

### Talker（System 1）

快速、直觉、对话式。用 LLM 的对话能力即时响应用户，从记忆中提取最新的用户画像来调整回复。不做复杂推理，只做流畅的交互。

### Reasoner（System 2）

慢速、深思、多步推理。从交互历史中提取用户信念（belief），用结构化格式（JSON/XML）维护，调用外部工具，制定多步计划。

两者通过共享记忆协作：Talker 收集实时用户输入写入记忆，Reasoner 从记忆中读取并更新信念状态。Talker 发现用户需求变复杂时，可以暂停等 Reasoner 完成推理再回复。

## 技术细节

**POMDP 框架**：用部分可观测马尔可夫决策过程建模 agent 与环境的交互。动作空间扩展到包含思考、信念和工具使用。

**层次化推理**：Reasoner 把复杂任务分解成子任务，逐层处理。

**显式信念建模**：Reasoner 维护结构化的用户信念状态，持续更新。这不是隐式的 context 理解，而是显式的、可检查的状态对象。

## 跟我们的关联

这个架构跟我们之前讨论的 FluidOS Talker/Worker 分离是同一个思路。今天我们刚移除了 Talker 的 presetTools，让职责更清晰——Talker 只负责对话，不做工具调用。

更深层的关联是跟记忆系统的：Reasoner 的显式信念建模跟我们图谱的 evidence 机制类似——不是让模型"记住"，而是把信念外化成可查询的结构化数据。

## 局限

论文用睡眠教练做 demo，场景相对简单。在更复杂的场景（如多文件代码重构）里，Talker 和 Reasoner 的边界会模糊——很多任务既需要快速响应又需要深度推理，不容易干净地分开。

## 关键启发

认知科学的框架可以直接指导 agent 架构设计。不是类比，是映射——人脑的双系统处理不同类型的认知任务，agent 也应该用不同的处理模式应对不同复杂度的请求。
