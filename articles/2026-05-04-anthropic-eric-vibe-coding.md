# Eric (Anthropic): 如何在生产环境负责任地 Vibe Code

Eric 是 Anthropic 研究员，Building Effective Agents 的合著者。这个 30 分钟演讲提出了一个完整的框架：如何把 vibe coding 从玩具项目带进生产环境。

## 核心框架

**1. 做 Claude 的 PM，不是写代码的人**

"Ask not what Claude can do for you, but what you can do for Claude." 他花 15-20 分钟准备一个 prompt——不是手写，而是跟 Claude 对话探索代码库、找文件、一起建计划，把所有上下文收集好再让 Claude 执行。像给新员工第一天布置任务一样，不能只说"做这个功能"。

**2. 聚焦叶节点，保护核心架构**

代码库是一棵树。叶节点（最终功能，没有其他东西依赖它们）可以放心让 AI 写，即使有 tech debt 也无所谓。但树干和主要分支（核心架构、会被扩展的部分）必须人类深度 review。Tech debt 是目前唯一没有好办法在不读代码的情况下验证的东西。

**3. 设计可验证性**

他们把一个 22,000 行的 PR 合入了 Anthropic 生产 RL 代码库。怎么做到的：系统设计成有"人类可验证的输入输出"，写压力测试跑长时间验证稳定性，不需要读每一行代码就能确信正确性。关键是设计系统时就考虑验证，而不是事后补。

**4. 拥抱指数增长**

AI 能完成的任务长度每 7 个月翻一倍。现在是 1 小时，明年是 1 天，后年是 1 周。如果还坚持读每一行代码，你就会成为瓶颈。编译器类比：早期程序员不信编译器会读汇编输出，但规模大了就必须信任系统。

## 实操建议

- 用 Claude Code 探索不熟悉的代码库，先建心智模型再动手
- 测试驱动：让 Claude 写端到端测试而非实现细节测试，先看测试再看代码
- compact 时机：像人类程序员"去吃午饭"的节点，把探索阶段的 100K token 压缩成几千 token 的计划文档
- Claude Code + VS Code 组合：Claude Code 做主要编辑，VS Code 做精确修改和 review

## 关键洞察

管理你不完全理解的实现，是文明一样古老的问题。CTO 管领域专家、PM 验收功能、CEO 抽查财务——都是同一个模式。软件工程师不习惯这个，因为我们一直是 IC，习惯理解全栈。但这正是需要改变的。
