# Harness Engineering 就是控制论

> "你不再转阀门。你在掌舵。" —— Norbert Wiener, 1948

## 三次相同的模式

读 OpenAI 的 harness engineering 文章时，作者 George (@odysseus0z) 意识到这个模式他见过三次：

**第一次：蒸汽机（1780s）**
James Watt 的离心调速器。之前工人手动调阀门，之后飞球机构自动感知转速、自动调节。工人没有消失——工作变了：从转阀门到设计调速器。

**第二次：Kubernetes**
你声明期望状态——3 个副本、这个镜像、这些资源限制。控制器持续观察实际状态，有偏差就调和。工程师的工作从重启服务变成编写系统要调和的 spec。

**第三次：现在**
OpenAI 描述了不再写代码的工程师。他们设计环境、构建反馈环路、编纂架构约束——然后 agent 写代码。5 个月 100 万行，零手写。他们叫它 "harness engineering"。

Norbert Wiener 在 1948 年就给这个模式命名了：**控制论 (cybernetics)**，来自希腊语 κυβερνήτης——舵手。

## 为什么代码库是最后的堡垒

代码库有反馈环路，但只在底层：
- 编译器闭环语法
- 测试套件闭环行为
- Linter 闭环风格

这些都是真正的控制论控制——但只对机械可检查的属性有效。编译通过？测试通过？符合规则？

更高层的判断——这个改动是否符合系统架构？这个方向对不对？这个抽象会不会随着代码库增长造成问题？——没有传感器也没有执行器。只有人类能在这个层面操作。

**LLM 同时改变了两侧。** 它们能在人类曾经独占的层面感知——也能在同一层面行动：重构模块、重新设计不一致的接口、围绕真正重要的契约重写测试套件。反馈环路第一次能在重要决策的层面闭合。

## 校准传感器和执行器

让基本反馈环路工作——agent 能跑的测试、可解析的 CI 输出、指向修复方案的错误信息——这是入场门票。Carlini 用 16 个并行 agent 构建 C 编译器时证明了这点："我大部分精力花在设计 Claude 周围的环境——测试、环境、反馈。"

更难的是用**你系统特有的知识**校准传感器和执行器。大多数人卡在这里，然后怪 agent。

"它总是做错。它不理解我们的代码库。"——诊断几乎总是错的。Agent 失败不是因为缺能力，而是因为它需要的知识——什么是"好"、哪些模式你的架构鼓励、哪些要避免——**锁在你脑子里，你没有外化出来**。Agent 不会通过耳濡目染学习。你不写下来，它第 100 次跑和第 1 次犯一样的错。

OpenAI 发现了这点：他们每周五花 20% 时间清理 "AI slop"——直到他们把标准编码进 harness 本身。

## 唯一的出路

这些实践——文档、自动化测试、编纂架构决策、快速反馈环路——一直都是正确的。过去三十年每本工程书都推荐。大多数人跳过它们，因为跳过的代价缓慢而弥散：质量逐渐下降、痛苦的入职、悄悄复合的技术债。

Agentic engineering 让代价变得极端。跳过文档，agent 忽略你的惯例——不是一个 PR，而是每个 PR，以机器速度，全天候。跳过测试，反馈环路根本无法闭合。跳过架构约束，漂移的速度超过你能修复的速度。

**陷阱在这里：你不能用 agent 来清理混乱，如果 agent 不知道什么是干净的。** 没有校准，制造问题的机器也解决不了问题。

## 核心洞察

**Generation-Verification Asymmetry（生成-验证不对称）** —— P vs NP 的直觉在 LLM 上的经验验证。生成正确解比验证一个解更难。你不需要比机器实现得更好。你需要比它**评判得更好**：定义什么是"正确"、识别输出何时偏离、判断方向是否正确。

设计 Watt 调速器的工人再也没回去转阀门。不是因为他们不能。是因为那不再有意义了。
