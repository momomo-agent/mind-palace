# Sema Code Core: 语义化代码理解引擎

> 来源：Sema 项目
> 标签：AI, Develop, Tool

## 问题：LLM 理解代码的方式是错的

LLM 把代码当文本处理——逐 token 读取，用注意力机制建立关联。这在短代码片段上效果不错，但在大型代码库上会失败，因为：

1. 关键的依赖关系可能跨越数千行甚至不同文件
2. 同名变量在不同作用域有不同含义
3. 类型信息和接口约束是隐式的

人类开发者理解代码不是逐行读的——是通过 IDE 的"跳转到定义""查找引用""类型提示"来导航的。Sema 试图给 AI 同样的能力。

## 核心技术

Sema Code Core 构建代码的语义图（semantic graph）：
- 节点 = 符号（函数、类、变量、类型）
- 边 = 关系（调用、继承、实现、导入、类型约束）

这个图不是 AST（抽象语法树）——AST 是语法结构，Sema 的图是语义结构。AST 告诉你"这里有一个函数调用"，Sema 告诉你"这个函数调用了那个模块的那个方法，参数类型是 X，返回类型是 Y"。

## 跟 RAG 的对比

传统 code RAG：代码 → 切块 → embedding → 向量搜索 → 返回相似块
Sema：代码 → 语义图 → 图查询 → 返回相关符号 + 完整上下文

区别在于"相似"vs"相关"。向量搜索找语义相似的代码块，Sema 找结构相关的符号。当你修改一个接口时，你需要的不是"长得像这个接口的代码"，而是"所有实现了这个接口的类"。

## 跟我们的关联

Sema 的语义图跟我们的记忆图谱是同构的：
- 我们的图谱：概念节点 + 关系边 + evidence
- Sema 的图：符号节点 + 依赖边 + 类型信息

两者都是用图结构来补充纯文本/向量搜索的不足。我们的 recall.js 用 1-hop 图扩展来找相关概念，Sema 用图遍历来找相关符号。

这也验证了 RAG-is-broken 文章的论点：对有结构的数据，图查询 > 向量搜索。

## 启发

代码理解的未来不是更好的 embedding，而是更好的结构化表示。语义图让 AI 能像人类开发者一样"导航"代码，而不是像搜索引擎一样"检索"代码。
