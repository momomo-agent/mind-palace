# RAG 对代码无效：为什么 Coding Agent 用 grep 而不是向量数据库

> 来源：MindStudio 分析 + 行业实践
> 标签：AI, Develop

## RAG 的前提假设

RAG（Retrieval-Augmented Generation）在 2022-2023 年是 AI 应用的标准架构：把数据切块、向量化、存进向量数据库，查询时找最相似的块塞进 context。

这套方案在几个前提下成立：
- 数据太大放不进 context
- 相关信息跟查询语义相似
- 切块后每块能独立理解
- embedding 能准确捕捉用户意图

对客服工单、知识库、文档搜索，这些前提基本成立。对代码，全部崩塌。

## 代码不是文档，是依赖图

代码的本质是一张依赖图——import、函数调用、类型定义、继承关系。第 200 行的函数可能依赖第 10 行的接口和另一个文件的工具函数。

向量 embedding 擅长捕捉语义相似性（"dog"和"puppy"距离近），但捕捉不了结构关系。一个调用 `processPayment()` 的代码块和定义 `processPayment()` 的代码块，在向量空间里可能距离很远——因为它们长得不像。

### 切块困境

代码切块面临两难：切太小丢失上下文，切太大引入噪音。散文可以摘要，但一个 50 行的函数通常不能——你需要完整的函数体 + 签名 + 类型定义才能理解它。

### 检索失败是灾难性的

在 Q&A 系统里，RAG 漏掉一个文档，用户得到不完整的答案——烦人但可恢复。

在 coding agent 里，漏掉一个关键函数定义或类型结构，agent 会自信地写出调用不存在方法的代码，然后在多个步骤里持续犯错。失败模式是沉默的、累积的。

## Coding Agent 实际用什么

2025-2026 年最成功的 coding agent（Claude Code、Cursor、Devin）收敛到了同一套模式：

1. **File tree inspection**：先看目录结构理解项目组织
2. **grep / ripgrep**：精确、确定性、带文件名和行号
3. **Import chain following**：顺着 import 找依赖
4. **AST 分析**：理解代码结构而非文本相似性

这就是有经验的开发者接手陌生代码库时做的事——不会去查向量数据库，而是看目录、搜关键词、跟 import。

## 跟我们记忆系统的关联

我们的记忆检索系统（recall.js）用的是 BM25 + bge-m3 向量 + RRF 融合 + 1-hop 图扩展。这个设计选择现在看来是对的：

- BM25 提供精确匹配（类似 grep 的确定性）
- 图扩展提供结构关系（类似 import chain following）
- 向量搜索作为补充而非主力

纯向量搜索的 seed recall 只有 32%，加上 BM25 和图扩展后到了 97.6%。这跟 coding agent 的经验完全一致——结构关系比语义相似性更重要。

## RAG 没死，但适用范围比想象的窄

RAG 在非结构化文本搜索（客服、知识库、法律文档）仍然有效。但对任何有结构关系的数据——代码、配置、API 文档——grep + 结构分析是更好的选择。
